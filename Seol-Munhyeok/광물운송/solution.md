## 1. 격자 문제처럼 보이지만 경로 문제인지 먼저 판단
* 상하좌우 이동, 에너지, 충전 → 겉보기는 BFS/시뮬레이션
* 하지만 장애물 없음 + 이동 과정 자체 의미 없음
⇒ 중요한 것은 경로가 아니라 “도달 가능 여부”

## 2. 이동 가능 조건을 거리로 압축
* 두 캠프 간 이동 가능 조건: 맨해튼 거리가 L 이하
⇒ BFS 없이 맨해튼 거리로 O(1) 판정 가능

## 3. 문제를 그래프로 변환
* 노드 = 베이스캠프
* 간선 = 거리 ≤ L

## 4. 핵심 통찰: “연결되어 있으면 전부 방문 가능”
* 캠프 도착 시 에너지 초기화
* 재방문 가능
* 채굴은 1번만

## 5. 문제를 “연결요소 문제”로 재정의

* 각 연결요소마다 유지해야 할 것:

  * sum[root] → 총 광물량
  * **best[root] → 우선순위 최상 캠프** (이거를 떠올리지 못했음. 각 연결요소 마다 우선순위 기준에 맞는 캠프를 저장하면 됨)

⇒ 이제 문제는:
“연결요소를 관리하면서 조건 만족하는 대표 캠프 찾기”

⇒ 결론:
한 캠프에서 시작하면 같은 연결요소 전체를 수집 가능

## 6. 동적 연결 → DSU 사용
* 캠프 추가 시:
  * 기존 캠프들과 연결 여부 판단
  * 연결되면 union
* union 시 관리:
  * sum 합치기
  * best 갱신

## 7. 여기서 1차 시간복잡도 문제 발생
naive
```
for (모든 기존 캠프) {
    거리 검사
}
```
⇒ O(N²) 위험

## 8. “모든 캠프를 볼 필요가 있는가?” 질문
* 핵심 통찰:
  * 거리 ≤ L 이므로 멀리 있는 캠프는 절대 연결 안됨

## 9. 공간 분할 (버킷) 도입

* 조건: `N <= 30L`
⇒ 힌트:
좌표를 L 단위로 나누면 버킷 수가 작다

## 10. 버킷 정의
```
br = row / L;
bc = col / L;
```
⇒ 공간을 L×L 크기 블록으로 나눔

## 11. 왜 3×3만 보면 되는가
버킷 한 칸 크기 = L
버킷 차이가 2 이상이면
최소 거리 > L 가능성 높음

⇒ 연결 가능성 있는 건:
자기 포함 주변 3×3 버킷

## 12. 버킷의 역할 (매우 중요)

* 버킷은 “답을 구하는 구조”가 아니라 “후보를 줄이는 구조”
* DSU → 연결요소 관리 (정답 로직)
* 버킷 → union 대상 찾기 최적화

## 13. addBaseCamp 흐름
1. 새 노드 생성
2. 자기 정보 저장
3. DSU 초기화
4. 자기 버킷 계산
5. 주변 3×3 버킷 탐색
6. 거리 ≤ L이면 union
7. 자기 버킷에 삽입
8. `sum[find(idx)]` 반환

## 14. findBaseCampForDropping 흐름
1. 모든 루트 순회
2. `sum[root] >= K` 필터링
3. 각 루트의 `best[root]` 후보 선택
4. 전체 우선순위 비교
5. `id[ans]` 반환
