## 풀이 전략
* `expand()` 연산 수행 시 가장 시간이 오래걸리는 도로를 찾는 연산이 필요함 -> TreeSet 또는 PriorityQueue 사용 고려
* `expand()` 연산에 의해 도로의 가중치가 자주 변함. update가 자주 일어나는 상황에서 구간합을 구하는 쿼리(`calculate()`)가 자주 일어남 -> Segment Tree 사용 고려
*  `divide()` 는 처음에 쿼리 횟수가 300임에 주목하여 브루트포스 접근을 고려하였으나 최악의 경우 10000C5000 의 연산 횟수 발생 -> 시간복잡도를 줄일 풀이법을 떠올리지 못했음
  * `divide()`는 풀이법을 검색해보았음. (이분 탐색 + 그리디 판정 사용)

## 핵심 자료구조
### 1. `population[]`
* 각 도시의 인구를 1-based로 저장했다.

### 2. `initDist[]`
* `i`번 도시와 `i+1`번 도시 사이의 초기 이동 시간을 저장했다.
* 문제 조건상 초기 이동 시간은 `population[i] + population[i+1]`

### 3. `bridgeCnt[]`
* 각 구간에 현재 몇 개의 다리가 있는지 저장했다.
* 초기값은 모두 1
* 다리를 추가할수록 해당 구간 이동 시간은 `initDist / bridgeCnt`

### 4. `TreeSet<Node> ts`
* 현재 가장 시간이 큰 구간을 빠르게 꺼내기 위해 사용했다.
* max 기준 내림차순
* 같으면 idx 오름차순
* 즉, pollFirst() 하면 항상 현재 가장 비효율적인 구간(=시간이 가장 큰)을 바로 가져올 수 있다.

### 5. `sumTree[]`
* 현재 각 구간 이동 시간들의 합을 관리하는 세그먼트 트리다.
* `calculate()`를 빠르게 처리하기 위해 사용했다.
* `expand()`에서 어떤 구간 이동 시간이 바뀌면 점 업데이트로 반영한다.

## 함수별 풀이 설명

### `void init(int N, int mPopulation[])`

#### 역할
초기 도시 정보와 다리 정보를 세팅한다.

#### 처리 내용
- 도시 수 저장
- `population[]` 1-based로 복사
- 인접 도시 간 초기 이동 시간 `initDist[i] = population[i] + population[i+1]` 계산
- 각 구간의 다리 수를 `1`로 초기화
- 각 구간 정보를 `TreeSet`에 삽입
- 세그먼트 트리 초기화

#### 의도
이후 연산에서
- 최대 이동 시간 구간 찾기
- 특정 도시 구간 이동 시간 합 구하기

를 빠르게 수행할 수 있도록 초기 상태를 구성한다.

---

### `private int initSumTree(int node, int start, int end)`

#### 역할
초기 구간 이동 시간 배열(`initDist`)을 기반으로 세그먼트 트리를 만든다.

#### 방식
- 리프 노드면 해당 구간 이동 시간 저장
- 아니면 좌우 자식의 합으로 현재 노드 값 구성

#### 의도
`calculate()`에서 구간 합을 `O(log N)`에 구하기 위한 준비 단계다.

---

### `private int querySum(int node, int nodeL, int nodeR, int queryL, int queryR)`

#### 역할
특정 다리 구간의 이동 시간 합을 구한다.

#### 방식
세그먼트 트리의 기본 구간 합 쿼리 방식이다.

- 완전 미포함 → `0`
- 완전 포함 → 현재 노드 값 반환
- 부분 겹침 → 왼쪽/오른쪽 자식으로 분할 탐색

#### 시간복잡도
`O(log N)`

---

### `private void updateSumTree(int node, int nodeL, int nodeR, int idx, int diff)`

#### 역할
어떤 다리 구간의 이동 시간이 변했을 때 세그먼트 트리에 반영한다.

#### 방식
- `idx`가 포함된 구간만 따라 내려가며
- 해당 노드들에 `diff`를 누적 반영한다.

#### 사용 시점
`expand()`에서 다리를 추가해 특정 구간의 이동 시간이 줄어들었을 때 호출한다.

#### 시간복잡도
`O(log N)`

---

### `int expand(int M)`

#### 역할
현재 이동 시간이 가장 큰 구간부터 `M`번 다리를 추가하고, 마지막으로 갱신된 이동 시간을 반환한다.

#### 핵심 아이디어
매번 전체 구간을 순회하면 비효율적이므로, `TreeSet`에 현재 구간 이동 시간을 정렬된 상태로 유지한다.

#### 처리 순서
1. `ts.pollFirst()`로 가장 큰 이동 시간을 가진 구간 선택
2. 해당 구간의 다리 수 증가
3. 새 이동 시간 계산  
   `changedDist = initDist[idx] / bridgeCnt[idx]`
4. 기존 값과의 차이 `diff`를 구해 세그먼트 트리 갱신
5. 새 값으로 다시 `TreeSet`에 삽입

#### 시간복잡도
- 한 번 갱신마다 `TreeSet O(log N)` + `Segment Tree O(log N)`
- 총 `O(M log N)`

#### 의도
“가장 비효율적인 구간부터 개선”해야 하므로 우선순위가 중요하고, 그 결과를 전체 이동 시간 합 계산에도 즉시 반영해야 해서 `TreeSet + 세그먼트 트리` 조합으로 처리했다.

---

### `int calculate(int mFrom, int mTo)`

#### 역할
두 도시 사이를 이동하는 데 필요한 총 시간을 구한다.

#### 핵심 포인트
도시 `a`에서 도시 `b`로 이동할 때 필요한 것은 도시 자체가 아니라 **사이의 다리 구간들**의 합이다.

예를 들어,
- 도시 `1 → 3`
- 필요한 다리: `(1-2)`, `(2-3)`

즉, 도시 구간 `[mFrom, mTo]`가 아니라 다리 구간 `[mFrom+1, mTo]`에 해당하는 합을 구해야 한다.

코드에서는 0-based 도시 번호를 1-based로 맞춘 뒤,  
`querySum(1, 1, bridgeN, mFrom, mTo - 1)` 형태로 처리했다.

#### 시간복잡도
`O(log N)`

---

### `private boolean canDivide(int mFrom, int mTo, int limit, int K)`

#### 역할
주어진 최대 허용 인구 `limit` 안에서 해당 도시 구간을 `K`개 이하의 선거구로 나눌 수 있는지 판정한다.

#### 방식
그리디하게 왼쪽부터 도시를 묶는다.

- 현재 선거구에 다음 도시를 넣어도 `limit` 이하이면 계속 포함
- 넘으면 새 선거구 시작
- 최종적으로 필요한 선거구 수 `cnt` 계산

#### 왜 그리디가 가능한가
최대 허용 인구가 `limit`로 고정되어 있을 때, 왼쪽부터 가능한 만큼 채우는 방식이 선거구 수를 최소화한다. 따라서 이 방식으로 구한 `cnt`가 `K` 이하이면 나누는 것이 가능하다.

#### 판정 기준
- `cnt <= K` 이면 가능
- 정확히 `K`가 아니어도, 더 잘게 쪼개서 `K`개로 맞출 수 있으므로 가능으로 본다.

---

### `int divide(int mFrom, int mTo, int K)`

#### 역할
도시 구간을 정확히 `K`개의 선거구로 나눌 때, 각 선거구 인구 합의 최댓값이 최소가 되도록 하는 값을 구한다.

#### 핵심 아이디어
이 값은 **최솟값을 찾는 전형적인 파라메트릭 서치 문제**다.

#### 탐색 범위
- `lo = 구간 내 단일 도시 인구의 최댓값`  
  → 이보다 작으면 어떤 도시도 담을 수 없음
- `hi = 구간 전체 인구 합`  
  → 최악의 경우 전부 한 선거구로 묶는 경우

#### 이분 탐색
- `mid`를 “선거구 최대 허용 인구”라고 가정
- `canDivide(..., mid, K)`가 가능하면 더 작은 값도 가능한지 왼쪽 탐색
- 불가능하면 더 크게 잡아야 하므로 오른쪽 탐색

#### 시간복잡도
- `canDivide()`가 `O(N)`
- 이분 탐색이 `O(log S)` (`S`는 인구 합 범위)
- 총 `O(N log S)`
