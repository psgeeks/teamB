# 병사관리

## 초기 접근법

### 1. 객채 배열

Soldier { int team, int score } 객체로 배열을 생성하고 List<Integer> team[6] 을 이용해서 팀별로 관리

객체를 계속 생성 + List에서 삭제하기 위한 IndexOf 연산 등으로 인한 병목현상 발생 (30s)

### 2. 속성별 배열 생성

객체가 아닌 team[], score[] 배열을 이용하여 접근 시간을 줄이고자 함.

결국 List 삭제 연산 및 updateTeam() 함수에서 모든 값을 갱신해주는 과정에서 병목현상 발생 (15s)

### 3. 지연 갱신

매 번 노드를 방문 및 갱신하는 것이 아닌, savedScore[6][6] 배열을 이용하여 특정팀의 특정 점수값을 누적으로 갱신하고, hire, update, bestSoldier 을 할 때마다 누적된 점수 반영하도록 함

계속해서 List 삭제 연산 및 갱신을 위한 반복 연산에서 병목현상 발생 (40s)

## 최종 로직 (AI 사용…)

설계 40m, 구현 30m, 디버깅 및 수정 2h를 사용하다 도저히 개선이 안 돼서 제미나이의 힘을 빌렸습니다. 다만 다른 분들과 다르게 class를 이용한 노드 구현이 아닌 모든 변수를 int 배열로 처리하였는데, 가독성과 이해에 있어서는 노드를 생성해주는 것이 쉬울 것 같네요..

**1. 변수 및 배열 설명**
이 코드는 객체(Object) 생성으로 인한 메모리 오버헤드와 가비지 컬렉션(GC) 시간을 줄이기 위해, 모든 데이터를 1차원 및 2차원 배열에 담아 관리합니다.

• **`int[] nxt`, `int[] id` (노드 풀 - Node Pool)**
    ◦ 연결 리스트의 노드 역할을 하는 배열들입니다.
    ◦ `nodeCnt`라는 변수가 새로운 노드가 생성될 때마다 1씩 증가하며 인덱스 역할을 합니다.
    ◦ `id[nodeCnt]`는 해당 노드가 어떤 병사(mID)의 정보인지를 저장합니다.
    ◦ `nxt[nodeCnt]`는 현재 노드 다음으로 연결된 노드의 인덱스 번호를 가리킵니다.

• **`int[] version`, `int[] v` (지연 삭제의 핵심)**
    ◦ `version[mID]`는 특정 병사(mID)의 **현재 최신 상태(버전)**를 의미합니다.
    ◦ `v[nodeCnt]`는 특정 노드가 **생성될 당시의 병사 버전**을 의미합니다.
    ◦ 데이터를 조회할 때 `version[mID] == v[nodeCnt]`인 경우에만 유효한(진짜) 데이터로 취급하고, 다르면 이미 해고되었거나 점수가 변경된 '유령(더미) 데이터'로 무시합니다.

• **`int[] team`**
    ◦ 병사(mID)가 현재 어느 팀에 속해 있는지를 저장합니다.

• **`int[][] head`, `int[][] tail` (팀/점수별 연결 리스트)**
    ◦ `head[mTeam][mScore]`는 `mTeam` 팀이면서 평판 점수가 `mScore`인 병사들이 모여있는 연결 리스트의 **시작 노드 인덱스**를 저장합니다.
    ◦ `tail[mTeam][mScore]`는 해당 리스트의 **마지막 노드 인덱스**를 저장합니다.
    ◦ `tail`을 따로 관리하는 이유는 리스트의 맨 끝에 새로운 병사를 추가하거나, 두 리스트를 하나로 합칠 때 탐색 없이 즉시(O(1) 시간 복잡도) 처리하기 위함입니다.

---

**2. 함수별 동작 과정 설명
`init()` : 초기화**
각 테스트 케이스 시작 시 호출되어 `nodeCnt`를 0으로 만들고 `head`와 `tail` 배열을 비워줍니다. 한 테스트 케이스 내에서 동일한 mID를 가진 병사가 여러 번 고용되는 경우는 없음이 보장되므로, `version` 배열은 매번 초기화할 필요 없이 계속 증가시켜 재사용하는 최적화가 적용되어 있습니다.

**`hire(int mID, int mTeam, int mScore)` : 병사 고용**
병사가 새로 들어오면 `version[mID]`를 증가시켜 새로운 상태임을 표시합니다. 그리고 해당 병사의 소속팀을 `team[mID]`에 저장합니다. 이후 새로운 노드를 하나 할당(`nodeCnt++`)하여 병사의 ID와 현재 버전을 기록한 뒤, `[mTeam][mScore]`에 해당하는 연결 리스트의 맨 끝(`tail`)에 덧붙여줍니다.

**`fire(int mID)` : 병사 해고**
배열이나 연결 리스트를 뒤져서 병사를 지우지 않습니다. 단순히 `version[mID]++` 를 통해 병사의 최신 버전을 올려버립니다. 이렇게 하면 연결 리스트 어딘가에 남아있는 이 병사의 노드는 생성 당시의 옛날 버전(`v[nodeCnt]`)을 가지고 있으므로, 나중에 검색할 때 자동으로 무시(지연 삭제)됩니다.

**`updateSoldier(int mID, int mScore)` : 개별 병사 점수 변경**
해고와 마찬가지로 기존 점수가 들어있는 노드를 찾아가서 수정하지 않습니다. `version[mID]++`를 통해 기존 노드를 무효화시킨 뒤, 변경된 새로운 점수(`mScore`)에 해당하는 연결 리스트에 **새로운 노드를 생성해서 추가**해 버립니다.

**`updateTeam(int mTeam, int mChangeScore)` : 부대 전체 점수 변경**
이 함수의 호출 횟수는 100,000 이하이므로 무조건 빠르게 동작해야 합니다. 팀원 개개인의 점수를 바꾸는 대신, **연결 리스트 덩어리 자체를 옮겨 붙이는 방식**을 사용합니다.

• 점수가 오를 때(`mChangeScore > 0`): 5점부터 1점 순으로 역순 탐색합니다. (1점부터 탐색하면 점수가 오른 리스트가 또 탐색되어 점수가 중복해서 오르는 대참사가 발생합니다.)
• 점수가 내릴 때(`mChangeScore < 0`): 1점부터 5점 순으로 탐색합니다.
• 이동할 타겟 점수(`nextS`)를 계산할 때, '변경 전 평판 점수 + mChangeScore'가 5보다 클 경우 평판 점수를 5로 변경합니다.
• 마찬가지로 '변경 전 평판 점수 + mChangeScore'가 1보다 작을 경우 평판 점수를 1로 변경합니다.
• 기존 리스트의 `tail`을 타겟 점수 리스트의 `head`에 연결해 주는 작업만 하므로, 병사가 몇 명이든 관계없이 순식간에 점수 이동이 완료됩니다.

**`bestSoldier(int mTeam)` : 최고 에이스 검색**
가장 높은 점수부터 찾아야 하므로 5점부터 1점 순으로 내려가며 연결 리스트를 탐색합니다.
• 리스트를 순회(`curr = nxt[curr]`)하면서 `v[curr] == version[cId]` 인지 확인하여 진짜 유효한 노드만 검사합니다.
• 평판 점수가 가장 높은 병사가 여러 명일 경우, 고유번호가 가장 큰 병사의 고유번호를 반환해야 하므로 `cId > maxId` 일 때 최댓값을 갱신해 줍니다.

• 가장 높은 점수 대역에서 유효한 병사를 한 명이라도 찾았다면, 아래 점수는 볼 필요가 없으므로 즉시 `maxId`를 반환하고 종료합니다.

---

결론적으로 각각의 ID에 대응하는 병사 노드를 생성해서 상태 및 prev, next를 관리하는 것이 아닌 매 연산에 대한 노드를 생성하고 version을 이용해 유효한 노드들(fire되지 않았으면서 가장 최신 버전)에 대해 새로운 연산 및 출력을 하도록 하는 코드입니다.

`bestSoldier`을 제외한 모든 연산이 $O(1)$의 시간 복잡도를 가지고, `bestSoldier` 의 경우 최악일 때 $O(N)$의 시간 복잡도를 가집니다.
